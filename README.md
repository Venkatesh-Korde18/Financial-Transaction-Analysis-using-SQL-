# Financial Transactions Analysis with SQL and DuckDB — 13M+ Row Dataset

## 📌 About
Hands-on SQL practice on a real-world financial transactions dataset 
containing over 13 million records. Unlike tutorial databases, this 
data is large, messy, and unpredictable — much closer to what you 
encounter in actual fintech and banking environments.

I chose DuckDB as the query engine specifically because of its ability 
to handle large-scale local files efficiently, without needing a full 
server setup. The focus is on understanding transaction behaviour, 
spotting data quality issues, and practising aggregation patterns 
relevant to fintech data roles.

## 💾 Dataset
The raw file exceeds 13M rows and is too large to host here directly.

- **Source:** [Financial Transactions Dataset on Kaggle](https://www.kaggle.com/datasets/transactions-fraud-datasets/transactions-data)
- **Size:** 13M+ rows
- **Domain:** Financial transactions / Banking

## 🛠 Tools Used
- **DuckDB** — analytical query engine for large-scale local data
- **SQL** — primary analysis language
- **Python / Pandas** — data loading and preprocessing
- **Google Colab** — development environment

## 🔍 Questions Explored
- What are the most common reasons transactions get rejected?
- How are rejections distributed across transaction types?
- What does overall transaction volume and value look like?
- Where are the data quality gaps in the dataset?
